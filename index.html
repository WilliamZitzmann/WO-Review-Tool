Overview

§1
v0.6.0

Will's WO Review Tool

A browser bookmarklet that injects a fixed right-side panel into Maximo work order pages, reads live DOM field values, evaluates configurable JS rules, and displays pass/fail/warn/na status in grouped tiles.

The tool injects a fixed 360px panel on the right side of any Maximo work order page. It traverses iframes to read field values and table data directly from the live DOM, evaluates JavaScript rule formulas, and displays results in grouped, collapsible tiles. The scan system automatically navigates Maximo tabs and dialogs to collect data from multiple areas of the WO, then returns to the WO tab. All configuration is persisted in browser localStorage.

LocalStorage Keys

KeyContents
__wo_rules_config Groups, rules, field registrations, settings (return message config, hotkey, message overrides)
__wo_scan_config Scan targets — tabs/dialogs to visit, wait conditions
__wo_vars_config Variables — named computed expressions
__wo_field_config Registered field mappings (tab, label, DOM id)
__wo_group_state Per-group collapsed/visible state and ordering

The Panel

§2

The panel is a fixed right-side overlay, 360px wide, that pushes page content left so nothing is obscured. It is independently scrollable and persists across Maximo tab navigation within the same page session.

Status Bar

The status bar below the header displays the current scan state. Possible values:

  • Scanning… — scan in progress
  • Scan complete HH:MM:SS AM/PM — last successful scan time
  • Reading WO tab… — initial data capture from WO tab

Scan Log

A collapsible area below the status bar shows per-step results after each scan. It is scrollable up to 80px tall. Each line shows the target name and its result:

Log EntryMeaning
Target: OK (Xms)Target captured successfully in X milliseconds
Target: TIMEOUT waiting for "text" (Xms)waitFor text never appeared within the timeout window
Target: FAILEDAn error occurred during navigation or capture
Target: skipped (condition false)The target's condition formula returned false; target was skipped

Group Tiles

Each tile card has the following structure:

Header bar
Background #252525. Contains: colored status dots (one per ruleRef), title in bold, optional ⓘ tooltip icon (teal), optional inline header message (colored by rule status), expand/collapse button (▶/▼), × hide button.
Banner message
Optional blue-tinted box shown at the top of the expanded group body. Static text configured per group.
Field rows
Captured field values with labels, arranged in configurable rows. Multiple fields can share a row with optional % width.
Variable fields
Variable computed values rendered as labelled field rows with a teal (var) tag next to the label.
Linked table
Shows captured table rows inline. Has a ⚙ Cols button to toggle column visibility.
Rule status blocks
Small colored circle + rule label + status label. Sub-messages (bulleted, colored by status) appear below for fail/warn rules with configured messages.

Rule Status Blocks

Inside expanded tiles, each rule referenced by the group shows a status block:

  • Small colored circle matching the rule result color
  • Rule label text
  • Status label: ✓ OK / ✗ Failed / ⚠ Warning / — N/A
  • Sub-messages (bulleted, colored by status) appear below the status line for fail/warn rules that have configured messages

Bottom Controls

Dark background (#222). Shows assembled return message text, or (No failed rules — return message will appear here) in muted gray when all rules pass. Has a 📋 clipboard copy button.

Red. Full-width flex 1. Opens Route Workflow dialog, selects "Return Work Order", fills memo with return message.
✓ Approve button
Green. Full-width flex 1. Opens Route Workflow dialog, selects "Complete Review", submits.

Full-width button. Reveals all tiles previously hidden with the × button.

Reordering Groups

Groups can be dragged by their header bar to reorder. The new order is saved automatically to __wo_group_state.

Rules

§3

Rules are JavaScript formulas evaluated against cached field and table data. Each rule returns a status value that drives the panel display and return message assembly.

Return Values

Return ValueStatusColor
truePass Green
falseFail Red
'warn'Warning Orange
'na'Not Applicable Yellow
throws exceptionError Purple

Rule Properties

PropertyTypeDescription
idstringUnique identifier for the rule
labelstringDisplay name shown in panel and used as fallback message
formulaJS stringThe rule logic. Must return true, false, 'warn', or 'na'
pass / fail / warnobjectOne message config per status: { short, long, returnMode, returnCustom } (pass has no returnMode/returnCustom — only fail/warn contribute to the return message)
  .shortstringOne-line message shown in the collapsed group header. Supports {{expr}}
  .longarrayList of { condition, msg } entries shown as a bullet list in the expanded tile. Every entry whose condition is blank or truthy is shown
  .returnModestring'none' / 'short' / 'long' / 'custom' — what this status contributes to the copied return message
  .returnCustomstringText used when returnMode is 'custom'. Supports {{expr}}

All three statuses (Pass/Fail/Warn) use the identical { short, long } shape, edited inline in each rule's collapsible Pass/Fail/Warn section in Setup → Rules — there's no separate "return message" screen anymore.

Helper Functions

FunctionDescription
F(field)Get a field value by "Tab :: Label" key. Also resolves by suffix — e.g. F('Status') matches the first registered field ending with ' :: Status'
T(table)Get all rows of a captured table as an array of objects (keyed by column header)
rowCount(table)Number of rows in a captured table
col(table, colName)Array of values from a specific column across all rows
has(table, colName, value)Returns true if any row has the given value in the specified column
hours(str)Parse a HH:MM or decimal string to a numeric hours value
hoursBetween(a, b)Hours between two datetime strings (format: DD/MM/YYYY HH:MM)
oneOf(val, arr)Returns true if val is in the array arr
contains(text, pattern)Regex test — returns boolean
matches(text, pattern)Returns an array of unique regex matches
isEmpty(v) / notEmpty(v)Empty / not-empty check (handles null, undefined, empty string)
maxLaborHours(tableTitle, nameCol, hoursCol)Returns the maximum total hours attributed to any single person in the specified labor table
V(id)Get a Variable's computed value by its ID or label

{{expr}} Interpolation

Embed dynamic values in any message string (a Short field, a Long entry's message, or a return message's Custom text) using {{expr}}.

  • The expression is evaluated at render time against the same cached data as rules
  • Do not use return inside {{...}} — for multi-statement expressions, wrap in an IIFE:
// Single expression — OK
"Lot: {{F('Lot Number')}}"

// Multi-statement — use IIFE
"Hours: {{(function(){ var h=hoursBetween(F('Actual Start'),F('Actual Finish')); return h?h.toFixed(1):'?'; })()}}"
Note: If {{expr}} renders as ? in the panel, the expression threw an error. Use the Debug Dump in Settings to verify field key names.

Conditional Messages (Long list entries)

Each entry in a Pass/Fail/Warn "Long" list has its own separate Condition and Message input — added with the "+ Add message" button in Setup → Rules:

Condition blank
The message is always included when the rule reaches that status.
Condition set
A JS formula (same rules as the main formula box) — the message is only included if it's truthy.
Note: Condition and message are separate fields, so a condition that references a qualified field key like F('Work Order :: Status') is completely safe — there's no delimiter to collide with. (Older versions parsed condition :: message out of a single line, which broke exactly on inputs like this; that's why this changed.)

Code Examples

Field check (with na guard)

var lot = F('Work Order :: Production Run Lot #');
if (!lot) return 'na';
return lot.length > 3;

Duration vs actual hours

var d = hours(F('Duration'));
var a = hoursBetween(F('Actual Start'), F('Actual Finish'));
if (!d || !a) return 'na';
return d <= a;

Warn on emergency work type

var wt = F('Work Type');
if (wt === 'EM') return 'warn';
return true;

Table row count check

return rowCount('Labor') > 0;

Using a variable

var total = parseFloat(V('total_labor_hours'));
if (isNaN(total)) return 'na';
return total >= 0.5;

Variables

§4

Variables are named computed expressions that return a display value (string, number, etc.) rather than a pass/fail status. They are evaluated before rules after each scan, making their values available for rules, group header messages, and field row display.

Creating Variables

Variables are created in Setup → Variables. Each variable has three properties:

ID
Unique string identifier. Used in V('id') calls.
Label
Display name. Can also be used in V('label') calls.
Formula
JS expression that must return a value of any type. All rule helper functions are available.

Using Variables

UsageHow
In rule formulasV('label') or V('v_idstring')
As group header inline messageSet the group's headerMsg type to 'variable' in Setup → Groups
As field rows in group bodyAdd to the group's Variable Fields list in Setup → Groups. Displayed with teal label and (var) tag.

Example Formulas

Work type and status summary

return F('Work Order :: Work Type') + ' — ' + F('Work Order :: Status');

Total labor hours from table

var rows = T('Labor');
var total = 0;
rows.forEach(function(r) {
  total += hours(r['Regular Hours'] || '0');
});
return total.toFixed(2) + ' hrs';
Tip: Variables are evaluated in the order they are defined. A variable can reference a previously defined variable via V(), but not one defined after it.
